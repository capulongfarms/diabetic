Data Flow — What happens on each arrow

①
Developer → GitHub  (git push) Claude Code edits index.html locally, then commits and pushes to the GitHub repository using the polcapulong-capulongfarms account.
②
GitHub → GitHub Pages  (auto-deploy) GitHub detects a new commit on main and automatically rebuilds the Pages site. Takes ~2–5 minutes. No manual step needed.
③
GitHub Pages → Browser  (page load) When the owner opens the URL, GitHub Pages serves the single index.html file (with all embedded CSS and JavaScript) over HTTPS. PIN login is required before data is shown.
④
Firestore → Browser  (read on startup) After PIN login, dbAll(name) fetches all docs from 4 collections in parallel: readings, tests, wifesum, and wifev. This is the live data. If Firestore is unreachable, the app falls back to localStorage automatically.
⑤
Browser → Firestore  (write on every change) When the owner adds, edits, or deletes a record, dbAdd(), dbSet(), or dbDel() writes to Firestore. All write operations require PIN re-entry via _guardedAsync(). Firestore is the single source of truth.
⑥
Browser ↔ localStorage  (fallback) localStorage stores the PIN hash, user preferences (names, glucose/BP thresholds, font size), and a Firebase config override. If Firestore is unreachable, all 4 collections fall back to localStorage automatically using the same read/write functions.

Step-by-step Sequences

When the owner opens the app

1
Browser downloads index.html from GitHub Pages CDN (includes all CSS, JS, and Chart.js / XLSX loaded from CDN)
2
PIN login screen appears — SHA-256 hash of entered PIN is compared against stored hash. 3 wrong attempts → 30-second lockout.
3
PIN matches → session token written to sessionStorage. App shell loads.
4
dbAll() runs for all 4 collections — readings, tests, wifesum, wifev — fetched from Firestore in parallel.
5
Firestore offline: falls back to localStorage copies. App still fully functional with last known data.
6
Dashboard fully loads — Summary panel renders stats and 30-day trend chart. All 8 panels become navigable.

When the owner saves a record

1
Owner opens a form (new reading, lab test, spouse visit, etc.) and fills in the data
2
On submit, _guardedAsync() intercepts — PIN modal appears to re-verify identity before any write proceeds
3
PIN confirmed → dbAdd() or dbSet() writes the record to Firestore
4
If Firestore write fails, the same function writes to localStorage instead — data is never lost
5
Toast notification: "Saved!" — dashboard re-renders with updated data immediately

When the app is updated

1
Claude Code edits index.html on the developer workstation
2
git commit + git push to GitHub repository
3
GitHub automatically triggers a Pages deployment
4
~2–5 minutes later, new version is live
5
All health records in Firestore are untouched — readings, tests, and spouse data survive every update
6
Owner refreshes the browser — new app version loads instantly with all existing data

Data Structure — Firestore Collections & localStorage Fallback

Collection Firestore Path Written by localStorage Fallback Key Notes
Daily Readings users/{UID}/readings dbAdd() / dbSet() / dbDel() diab_readings One doc per reading entry. Fields: date, glucose (mmol/L), systolic, diastolic, weight (kg), meal description, medications, notes.
Lab Tests users/{UID}/tests dbAdd() / dbSet() / dbDel() diab_tests One doc per test session. Each session contains an array of criteria results (e.g. HbA1c, cholesterol, creatinine). Session date and lab name stored at the doc level.
Spouse Treatment Periods users/{UID}/wifesum dbAdd() / dbSet() / dbDel() diab_wifesum One doc per treatment period. Tracks condition, medication, dosage, and date range for the spouse/secondary tracked person.
Spouse Visit Records users/{UID}/wifev dbAdd() / dbSet() / dbDel() diab_wifev One doc per medical visit. Stores visit date, doctor, clinic, findings, and next appointment.
PIN & Preferences Not synced to Firestore Settings panel prefs, diab_pin Device-local only. prefs stores names, glucose thresholds (high/low mmol/L), BP thresholds (systolic/diastolic). diab_pin stores the SHA-256 PIN hash.
Firebase Config Override Not in Firestore Settings panel diab_firebase_config Optional per-device override of the hardcoded Firebase config. Useful if migrating to a different Firebase project.
Important: Firestore is the primary store. The fixed user UID (64oGBkwTKJOfCpGUW10eTrLdMgJ2) means all data is scoped to a single owner — this is by design for a private personal health app. localStorage is the fallback, not a cache: it is only read when Firestore is unreachable, not on every page load.

PIN Access Control — What Requires a PIN

Viewing data requires only a valid session (PIN entered on page load). Every write operation — add, edit, delete, import, export, backup, restore — re-prompts for the PIN via _guardedAsync(). This prevents accidental or unauthorized changes even if the browser is left open.

PIN on login only — read operations

📊View Summary stats and glucose trend chart
📋Browse, search, and filter readings table
📈View all chart visualizations (glucose, BP, weight)
🥗View diet analysis and meal rankings
❤️View blood pressure trends and averages
🧪View lab test sessions and results
👩‍⚕️View spouse treatment periods and visit records

PIN re-entry required — every write

➕Add new reading, lab test session, or spouse record
✏️Edit any existing record
🗑️Delete any record
📥Import CSV (readings, tests, spouse visits)
📤Export data to CSV or Excel workbook
💾Download JSON backup
♻️Restore from JSON backup
⚙️Save preferences (names, thresholds, font size)
🔑Change PIN
How the PIN session works: On successful login, a session token is written to sessionStorage. This survives page refreshes within the same browser tab but is cleared when the tab is closed — so the owner must re-enter the PIN on each new browser session. For individual write operations, _guardedAsync(fn) wraps the action: it shows the PIN modal, verifies the hash, then calls fn() only on success. Brute-force protection applies to both the login screen and mid-session PIN prompts: 3 wrong attempts → 30-second lockout.
