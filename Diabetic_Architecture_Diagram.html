Data Flow — What happens on each arrow

①
Developer → GitHub  (git push) Claude Code edits index.html locally, then commits and pushes to the GitHub repository using the capulongfarms account.
②
GitHub → GitHub Pages  (auto-deploy) GitHub detects a new commit on main and automatically rebuilds the Pages site. Takes ~2–5 minutes. No manual step needed.
③
GitHub Pages → Browser  (page load) When the owner opens the URL, GitHub Pages serves the single index.html file (with all embedded CSS and JavaScript) over HTTPS. The app loads immediately — no login gate. PIN is required only for write operations.
④
Firestore → Browser  (read on startup) On startup, dbAll(name) fetches all docs from 4 collections in parallel: readings, tests, wifesum, and wifev. Firestore is always tried first. If unreachable, the app falls back to localStorage automatically and shows a warning toast. The storage indicator updates to reflect which source is active.
⑤
Browser → localStorage + Firestore  (write-through on every change) When the owner adds, edits, or deletes a record, the local cache is always written first, then Firestore. All write operations require PIN re-entry via _guardedAsync(). If Firestore is unavailable, the record is safe in localStorage and the user is notified. Firestore is the single source of truth.
⑥
Browser ↔ localStorage  (write-through cache + prefs) localStorage is kept in sync at all times — not used as a last resort. Every successful Firebase read syncs the result to localStorage. Every write goes to localStorage first. It also stores the PIN hash and user preferences (names, glucose/BP thresholds, font size). If Firestore is unreachable, the app reads from this always-current local cache.

Step-by-step Sequences

When the owner opens the app

1
Browser downloads index.html from GitHub Pages CDN (includes all CSS, JS, and Chart.js / XLSX loaded from CDN)
2
Offline check — if navigator.onLine is false, a warning toast appears. Online/offline event listeners registered.
3
dbAll() runs for all 4 collections — readings, tests, wifesum, wifev — fetched from Firestore in parallel. App shell shows immediately.
4
Firestore offline: falls back to localStorage copies and shows a "Firebase unavailable" toast. Storage indicator updates. App still fully functional with last known data.
5
Security guards applied via _applySecurityGuards() — all write functions wrapped with PIN requirement. No race window.
6
Dashboard fully loads — Summary panel renders stats and 30-day trend chart. All 8 panels navigable. PIN required for every write.

When the owner saves a record

1
Owner opens a form — Add mode (no PIN anywhere) or Edit mode (PIN required once, on the Edit button click via editSelected*()).
2
Edit only: PIN was already collected when the Edit button was clicked. The modal Save button calls the unguarded save*() directly — no second PIN prompt.
3
Record written to localStorage first (instant, always succeeds)
4
Firestore write attempted — if it succeeds, both storages are in sync. If it fails, data is already safe in localStorage and a toast notifies the user.
5
Toast notification: "Saved!" — dashboard re-renders with updated data immediately

When the app is updated

1
Claude Code edits index.html on the developer workstation
2
git commit + git push to GitHub repository
3
GitHub automatically triggers a Pages deployment
4
~2–5 minutes later, new version is live
5
All health records in Firestore are untouched — readings, tests, and spouse data survive every update
6
Owner refreshes the browser — new app version loads instantly with all existing data

Data Structure — Firestore Collections & localStorage Fallback

Collection Firestore Path Written by localStorage Cache Key Notes
Daily Readings users/{UID}/readings dbAdd() / dbSet() / dbDel() data_{UID}_readings One doc per reading entry. Fields: date, glucose (mmol/L), systolic, diastolic, weight (kg), meal description, medications, notes.
Lab Tests users/{UID}/tests dbAdd() / dbSet() / dbDel() data_{UID}_tests One doc per test session. Each session contains an array of criteria results (e.g. HbA1c, cholesterol, creatinine). Session date and lab name stored at the doc level.
Spouse Treatment Periods users/{UID}/wifesum dbAdd() / dbSet() / dbDel() data_{UID}_wifesum One doc per treatment period. Tracks condition, medication, dosage, and date range for the spouse/secondary tracked person.
Spouse Visit Records users/{UID}/wifev dbAdd() / dbSet() / dbDel() data_{UID}_wifev One doc per medical visit. Stores visit date, doctor, clinic, findings, and next appointment.
PIN & Preferences Not synced to Firestore Settings panel prefs, sec_pin_h Device-local only. prefs stores names, glucose thresholds (high/low mmol/L), BP thresholds (systolic/diastolic). sec_pin_h stores the SHA-256 PIN hash. sec_lock and sec_attempts store brute-force state.
Firestore Security Rules Not a collection — Firebase-side rule firestore.rules (deployed via Firebase Console) N/A Locks Firestore access to users/64oGBkwTKJOfCpGUW10eTrLdMgJ2/* only. All other paths return permission-denied.
Important: Firestore is the single source of truth. The fixed user UID (64oGBkwTKJOfCpGUW10eTrLdMgJ2) scopes all data to a single owner — this is by design for a private personal health app. localStorage is a write-through cache — always kept in sync, not a last resort. Every successful Firebase read syncs to localStorage; every write goes to localStorage first then Firebase. If Firebase is unreachable, the app reads from this always-current local cache and a toast warns the user. Firestore Security Rules lock the database path so no other user can read or write the data.

PIN Access Control — What Requires a PIN

The app loads with no login gate — all data is immediately visible. Adding new records requires no PIN (the Add modal opens and saves freely). Edit, delete, import, export, backup, and restore operations re-prompt for the PIN via _guardedAsync() (SESSION_TTL = 0, perpetual mode). This prevents accidental or unauthorized changes even if the browser is left open.

How the PIN model works: There is no login gate — the app loads immediately. The modal Save button is never PIN-guarded: it calls save*() directly in both Add and Edit mode. For Add, no PIN is needed at any step. For Edit, the PIN is collected once when the Edit button is clicked (editSelected*() is guarded); Save does not prompt again. All other destructive operations (delete, import, export, backup, restore, preferences, change PIN) are individually guarded via _guardedAsync(fn): it shows the PIN modal, verifies the SHA-256 hash, then calls fn() only on success (SESSION_TTL = 0 — perpetual PIN mode, no session caching). If no PIN is set, only "Change Admin PIN" is allowed through; all other writes are blocked and redirect to Settings. Brute-force protection: 5 wrong attempts → 60-second lockout.
